01
// EXECUTIVE SUMMARY

الملخّص التنفيذي

منجِز منصة وكلاء ذكاء اصطناعي تساعد البنوك السعودية على مراجعة طلبات القرض العقاري. يمر كل طلب على خمسة وكلاء متخصصين، ثم يخرج النظام بتوصية أولية واضحة تراعي السياق المحلي والضوابط التنظيمية.

منجِز · تقييم القرض العقاري الأوّلي طبقة ذكية لمراجعة طلبات التمويل العقاري. تضم خمسة وكلاء متخصصين، ولكل وكيل نموذج LLM يناسب مهمته: الجدارة الائتمانية، التحقق من الهوية والأهلية القانونية، قدرة العميل والدعم، تقييم العقار والضمان، ثم التوصية النهائية. يتبادل الوكلاء السياق عبر بروتوكول A2A داخلي، ويصدرون توصية أولية مرتبة (موافقة / مشروطة / مراجعة بشرية / غير موصى) خلال أقل من دقيقتين، بصياغة عربية واضحة، وحدود صلاحيات معلنة، وسجل تدقيق يمكن الرجوع إليه.

Munjiz Mortgage AI is an intelligent review layer for Saudi mortgage applications. Five specialized agents — Credit, Identity & Legal, Capacity & Support, Collateral, and the Orchestrator — each use an LLM suited to the task, coordinate through an internal A2A protocol, and produce a structured pre-approval recommendation in under two minutes, in clear Arabic, with explicit authority limits and a traceable audit record.

// THE THESIS

تقييم طلب التمويل العقاري يجمع بيانات من مصادر مختلفة (SIMAH، أبشر/يقين، تحقق من جهة العمل، Sakani أو تثمين العقار)، ثم يراجع جدارة العميل وأهليته القانونية وقدرته الفعلية وضمانه. منجِز يرتب هذه المراجعة المتكررة في مسار ثابت، ويترك الحالات الحدية للمراجع البشري.

A mortgage application review pulls data from multiple sources — SIMAH, Absher/Yakeen, employer verification, Sakani or independent valuation — and weighs creditworthiness, legal eligibility, real affordability, and collateral. Munjiz turns that repeatable review into a consistent, structured path, leaving edge cases to a human reviewer.

02
// SYSTEM ARCHITECTURE

المعمارية النظامية

يمتد النظام من المتصفح إلى مزودات النماذج. كل طبقة لها مسؤولية واضحة، ويمكن اختبارها أو استبدالها دون إعادة بناء النظام كله.

الطبقات السبع · Seven Layers

  1. Browser Layer — Vanilla JavaScript + Alpine.js + Tailwind. عميل WebSocket مع dedupe بثلاث طبقات، ونظام ثيمات، وتجربة عربية RTL أولاً.
  2. Edge / Reverse Proxy — Nginx يخدم الملفات الثابتة، ويمرر HTTP إلى Daphne ASGI server، ويدعم ترقية اتصال WebSocket.
  3. Application Layer — Django 5 + Django Channels + DRF. يستقبل طلبات HTTP وإطارات WebSocket، ويدفع المهام الطويلة إلى Celery.
  4. Async Compute Layer — Celery workers (concurrency=4 لكل instance) تلتقط مهام مراجعة الطلبات من Redis queue وتشغّل AgentEngine مع سياق الطلب.
  5. Agent Engine — قلب النظام. AgentEngine.run_review_with_ai(application) يستدعي الوكلاء الخمسة بالتسلسل، وكل وكيل يستخدم LLM مختلفاً عبر OpenRouter API ويقرأ نتائج الفحوصات المناسبة لدوره (SIMAH, Absher, Sakani…).
  6. Data Layer — PostgreSQL 16 للبيانات الدائمة: المتقدّمون، الطلبات، المستندات، مراجعات الوكلاء، التوصية النهائية، وسجلات التدقيق. Redis يستخدم للـ message broker و channel layer.
  7. External LLM Surface — OpenRouter يعمل كواجهة تجميع واحدة ويمرر الطلبات إلى Anthropic / OpenAI / Alibaba (Qwen) / DeepSeek عبر API موحد ومتوافق مع OpenAI.

03
// HETEROGENEOUS MULTI-MODEL ARCHITECTURE

المعمارية متعددة النماذج

اختيار النماذج في منجِز ليس نسخة واحدة للجميع. كل وكيل يستخدم النموذج الأقرب لطبيعة دوره، مع موازنة بين جودة التفكير والتكلفة وثبات المخرجات.

لماذا ليس نموذجاً واحداً؟ · Why Not a Single Model?

الاعتماد على نموذج واحد يعني التنازل في جانب ما. Claude Opus قوي في الموازنة بين توصيات الموافقة والشروط لكنه مكلف. GPT-4o-mini سريع ومنضبط مع قواعد التحقق من الهوية والأهلية القانونية، لكنه ليس الأفضل في الصياغة العربية. Qwen3 80B ممتاز عربياً وفي حوار قدرة العميل، لكنه ليس الخيار الأول لحسابات SIMAH وDBR. DeepSeek V3.2 اقتصادي ومفيد في الاستدلال الرقمي على الجدارة الائتمانية، لكنه أقل مناسبة لتقييم العقار متعدد العوامل. لذلك يقوم التصميم على فكرة مباشرة: النموذج المناسب في المكان المناسب.

المعايير الثلاثة لاختيار النموذج · Three Selection Criteria

// 01

الملاءمة المعرفية

هل يثبت النموذج قوته في هذه المهمة تحديداً؟ Claude Opus لتوصية الموافقة النهائية، Sonnet لتقييم العقار وموازنة القيمة العادلة مع السعر المُعلَن، DeepSeek لحسابات الجدارة الائتمانية، Qwen للحوار العربي حول قدرة العميل، وGPT-4o-mini للتحقق المنضبط من الهوية والأهلية. الاختيار مبني على نتائج، لا على الانطباع العام.

// 02

التكلفة لكل طلب

سعر المليون توكن وحده لا يكفي للحكم. الرقم الأهم هو تكلفة مراجعة طلب واحد بحجم المدخلات الفعلي. تشغيل كل شيء على Opus يصل إلى ٠.٢٥$ للطلب، بينما التصميم المتعدد يقارب ٠.٠٨$، أي وفر يقارب ٧٠٪.

// 03

موثوقية الإخراج

الجودة وحدها لا تكفي؛ نحتاج مخرجات يمكن الاعتماد عليها في كل تشغيل. في الاختبار العملي، كان DeepSeek R1 يستهلك الميزانية في التفكير الداخلي ثم يعيد content فارغاً، لذلك استبدلناه بـ V3.2 instruct.

04
// AGENT-TO-AGENT PROTOCOL

بروتوكول A2A

يوضح الجزء التالي كيف يتبادل الوكلاء الرسائل والسياق، ولماذا يأتي كل وكيل في ترتيبه داخل سلسلة اتخاذ القرار.

المغلَّف الموحَّد · The Unified Envelope

كل تفاعل بين الوكلاء يمر عبر AgentMessageEnvelope، وهو dataclass موحد يجعل الرسائل مفهومة لكل أجزاء النظام، حتى لو كانت النماذج المستخدمة خلف الوكلاء مختلفة. 

# apps/agents/a2a_protocol.py
@dataclass
class AgentMessageEnvelope:
    from_agent: str                    # 'analytics' | 'operations' | 'customer' | 'queue' | 'orchestrator'
    content:    str                    # الرد الكامل للوكيل بالعربية
    message_type: str = 'update'      # analysis | update | suggestion | decision | alert
    priority:     str = 'medium'     # low | medium | high | critical
    to_agent:     Optional[str] = None  # None = broadcast لكل التبويبات
    to_broadcast: bool = True
    raw_data:     Optional[dict] = field(default=None)  # البيانات التي بَنى عليها الوكيل قراره

تسلسل المراجعة · The Review Sequence

الترتيب مقصود ويحاكي طريقة مدير ائتمان متمرس: نبدأ بالجدارة الائتمانية، ثم الهوية والأهلية القانونية، ثم قدرة العميل والدعم، ثم تقييم العقار/الضمان، وأخيراً التوصية النهائية.

01

سارة (الجدارة الائتمانية) · Sara (Credit)

تبدأ سارة لأن الجدارة الائتمانية هي بوابة الطلب. تقرأ بيانات SIMAH (الدرجة، التأخّرات، الالتزامات المفتوحة) وتحسب DBR بعد التمويل المطلوب، ثم تقدم حكماً رقمياً مع نسبة ثقة.

02

فهد (الهوية والقانوني) · Fahad (Identity & Legal)

يتحقق فهد من الهوية عبر أبشر/يقين، ويطابق راتب العميل مع جهة العمل، ويفحص أي قيود قانونية مثل إيقاف الخدمات أو حظر السفر. يضيف حالة الهوية، وتطابق صاحب العمل، وقائمة الحواجز إن وجدت.

03

خالد (قدرة العميل والدعم) · Khalid (Capacity & Support)

يفحص خالد القدرة الفعلية للعميل والإفصاحات الطوعية: اللغة، احتياجات الوصول، وليّ الشؤون القانوني، والحاجة إلى استشارة مستقلة. يضيف علامات الدعم اللازمة ويحدد الحاجة إلى ممثل قانوني.

04

نورة (تقييم العقار) · Nora (Collateral)

تقيّم نورة العقار والضمان. تختار مسار Sakani أو التثمين المستقل، وتقارن السعر المُعلَن بالقيمة العادلة، وتحسب LTV، وتضع إشارة عند وجود فجوة كبيرة.

05

المنسق · The Orchestrator

يأتي المنسق في النهاية لأنه صاحب التوصية. يستلم agent_inputs وفيها مخرجات الوكلاء الأربعة، ثم يصدر توصية مرتبة من أربعة احتمالات (موافقة أولية / مشروطة / مراجعة بشرية / غير موصى) مع الشروط والمخاطر المتبقية.

آلية تمرير السياق · Context Passing Mechanism

كل وكيل لاحق يستلم ما قاله من سبقه. مثلاً، عند استدعاء نورة، يحتوي user_message على:

{
  "application": { "id": "DEMO-APP-001", "requested_loan_sar": 850000, "requested_term_years": 25 },
  "applicant":   { "name_ar": "أبو محمد", "monthly_salary_sar": 22000, "employer_sector": "government" },
  "property":    { "source": "sakani", "city": "الرياض", "declared_price_sar": 950000 },
  "sara_review":   "📊 SIMAH 752 · DBR بعد التمويل 41٪...",
  "fahad_review":  "⚙️ الهوية مطابقة · صاحب العمل مؤكَّد · لا قيود...",
  "khalid_review": "💬 لا حاجة لوليّ شؤون · مستوى دعم اعتيادي...",
  // نورة الآن ترى أحكام زملائها قبل أن تقيّم العقار
}

صيغة الإخراج · Mandated Output Structure

لكل وكيل قالب إخراج داخل system prompt. القالب يترك مساحة للغة الطبيعية، وفي الوقت نفسه يضمن وجود structured JSON block قابل للقراءة الآلية والتدقيق البشري:

  • سارة: 📊 [SIMAH score] + [DBR after loan] + [verdict] + [مبرّر مختصر]
  • فهد: [حالة الهوية] + [تطابق صاحب العمل] + [قائمة الحواجز] + [تحذير إن وُجد]
  • خالد: [الحاجات المُفصَح عنها] + [قرار وليّ الشؤون] + [الموافقة الأخلاقية]
  • نورة: [مسار التقييم] + [القيمة العادلة] + [LTV] + [علم الانحراف]
  • المنسق: [تحليل موجز] ثم [توصية من أربعة] مع [شروط] و[مخاطر متبقّية للمراجع البشري]
// FUTURE: GOOGLE A2A STANDARD

البروتوكول الحالي بسيط عمداً. إذا احتجنا لاحقاً إلى workflows غير متزامنة، أو تنسيق بين فروع متعددة، أو agent discovery، يمكن توسيعه باتجاه معيار Google A2A الكامل دون تغيير كبير في شكل الرسالة الحالي.

05
// AGENT SPECIFICATIONS

مواصفات الوكلاء التفصيلية

لكل وكيل دور واضح، نموذج مناسب، مدخلات ومخرجات معروفة، وحدود صلاحية لا يتجاوزها.

01

سارة · Sara — الجدارة الائتمانية

MODEL: deepseek/deepseek-v3.2 · TEMP: 0.2 · COLOR: #FF1744 · TYPE: analytics

المسؤولية: تقرأ بيانات SIMAH وتاريخ الالتزامات، وتحسب DBR بعد التمويل المطلوب، ثم تصدر حكماً رقمياً (آمن / تحذير / خطر) مع نسبة ثقة. كل حكم تقدمه يكون مستنداً إلى رقم واضح.

المدخلات: applicant (راتب، التزامات شهرية، عمر)، application (مبلغ التمويل، المدة)، compliance snapshot من SIMAH.

المخرجات (structured JSON): simah_score_demo, dbr_after_loan_percent, verdict, late_payments, open_obligations.

عيِّنة برومبت نظامي:

- لا تخرجي بحكم لا يدعمه رقم
- استخدمي SIMAH score + DBR كركيزتين
- bands: SIMAH ≥750 آمن · 600-749 تحذير · <600 خطر
- bands: DBR ≤45٪ آمن · 46-55٪ تحذير · >55٪ خطر
- صياغة ردك: 📊 [الدرجة + DBR] + [التفسير] + [الحكم] + [مبرّر مختصر]
02

فهد · Fahad — الهوية والأهلية القانونية

MODEL: openai/gpt-4o-mini · TEMP: 0.3 · COLOR: #FF6D00 · TYPE: operations

المسؤولية: يتحقق من الهوية عبر أبشر/يقين، ويطابق راتب العميل مع جهة العمل المُعلَنة، ويفحص أي قيود قانونية. يعطي نتيجة واضحة، ويحوّل أي شك إلى المنسق.

المخرجات (structured JSON): identity_check_demo (passed / flagged / failed), employer_verification_demo, salary_match, legal_blockers_demo (list).

قواعد الصلاحيات:

  • الهوية مطابقة + جهة العمل مؤكَّدة + لا قيود ← يدفع للمراجعة العادية
  • أي علامة flagged أو حاجز قانوني ← يصعّد إلى المنسق مباشرة
  • فشل التحقق التام ← يوصي بـ not_recommended قبل وصول باقي الوكلاء

خالد · Khalid — قدرة العميل والدعم

MODEL: qwen/qwen3-next-80b-a3b-instruct · TEMP: 0.7 · COLOR: #2979FF · TYPE: customer

تدفق التنفيذ الكامل

من إدخال الطلب ورفع المستندات إلى ظهور التوصية النهائية على الشاشة، هذا هو مسار مراجعة الطلب داخل النظام.

المراحل الثمانية · The Eight Stages

01

الواجهة تُنشئ الطلب وترفع المستندات

يدخل موظف القروض بيانات المتقدّم والعقار في Stage 01، ويرفع المستندات (هوية، تعريف بالراتب، خطاب جهة العمل، كشف بنكي، بيان Sakani أو صك). بعدها تستدعي الواجهة:

POST /api/applications/                        # يُنشئ MortgageApplication بحالة 'draft'
POST /api/applications/<id>/documents/          # لكل مستند مرفوع
GET  /api/applications/<id>/compliance-preview/ # preview الفحوصات قبل التشغيل
02

المستخدم يضغط "بدء المراجعة" → POST /trigger

ترسل الواجهة POST /api/applications/<id>/trigger/. View الـ Django يحدّث application.status = 'under_review' ثم يرسل المهمة إلى Celery عبر run_review_task.delay(application_id)، ويعيد ack مباشرة بزمن استجابة أقل من 50ms.

03

Celery worker يلتقط مهمة المراجعة من Redis queue

تبني الـ task السياق من قاعدة البيانات عبر application.to_context(): applicant, application, property, screening_answers, documents. ثم تمرر السياق إلى AgentEngine.run_review_with_ai(application)، حيث يعمل الوكلاء الخمسة بالتسلسل عبر OpenRouter.

04

review_started broadcast

قبل تشغيل أي وكيل، يرسل الـ engine حدث lifecycle باسم review_started إلى group application:<id> في Redis pub/sub. كل تبويب مفتوح للطلب يستقبل الحدث ويمسح أي مراجعة قديمة، حتى لا تختلط الرسائل بين الطلبات.

05

حلقة الوكلاء الخمسة

لكل وكيل بالتسلسل (Sara → Fahad → Khalid → Nora → Orchestrator):

  • أ. تحديث agent.status = 'thinking'، ثم broadcast للواجهة حتى يظهر مؤشر التفكير.
  • ب. بناء user_message: سياق الطلب + مراجعات الوكلاء السابقين.
  • ج. استدعاء OpenAI(base_url=openrouter).chat.completions.create(...) ← POST لـ openrouter.ai.
  • د. فحص response.choices[0].message.content — إذا فارغ → raise LLMUnavailable.
  • هـ. حفظ AgentReview(content, structured, compliance_snapshot) في PostgreSQL.
  • و. بث الرسالة + الـ structured JSON إلى الواجهة.
  • ز. تحديث agent.status = 'done'، broadcast.
06

FinalRecommendation persistence + review_ended

بعد انتهاء المنسق، يستخرج الـ engine الـ structured من مراجعته ويحفظ صفاً واحداً في FinalRecommendation (recommendation + conditions_if_any + residual_risks_for_human_review). بعدها تتغيّر حالة الطلب إلى review_complete ويُبثّ review_ended.

07

المتصفح يستقبل ١٠–١٥ event عبر WebSocket

يمر كل event عبر MunjizBus، وهو EventTarget داخلي، بعد ثلاث طبقات dedupe: IIFE init guard، وseenMessageIds على مستوى الـ bus، وفحص data-msg-id على مستوى DOM. كل event يحمل agent_type وstructured JSON يغذّي حقول liveNumbers في الواجهة (sara.simah, fahad.id_status…).

08

الانتقال إلى Stage 03 — تقرير التوصية

عند استلام review_ended تنتقل الواجهة إلى Stage 03 وتنادي GET /api/applications/<id>/report/ لاسترجاع التقرير الكامل (FinalRecommendation + AgentReview list). بعدها يظهر الـ recommendation pill (موافقة أولية / مشروطة / مراجعة بشرية / غير موصى) مع قوائم الشروط والمخاطر المتبقية.

// RESILIENCE & MULTI-PROVIDER FAILOVER

الصمود والتعافي

يتعامل النظام مع الفشل كاحتمال وارد: يعيد المحاولة، ويتحقق من المخرجات، ويمكنه التحول بين مزودين عند الحاجة.

// LAYER 01

SDK Auto-retry

OpenAI SDK يعيد المحاولة تلقائياً عند rate limits (default: 2 retries) مع exponential backoff. هذا يعالج أغلب الفشل العابر دون تدخل من المستخدم.

// LAYER 02

Multi-Provider Failover

نفس الواجهة المتوافقة مع OpenAI متاحة عبر OpenRouter, Anthropic Direct, Groq, Azure OpenAI, Ollama-local. إذا فشل المزود الأساسي، ينتقل الطلب إلى المزود التالي حسب priority list.

// LAYER 03

Validated Output

كل رد يُفحص قبل قبوله: if not result.strip(): raise LLMUnavailable(...). أي محتوى فارغ أو غير صالح يفعّل retry policy تلقائياً.

قرارات هندسية موثَّقة · Documented Engineering Decisions

  • OpenRouter slug evolution: النموذج الأول (qwen-2.5-72b-instruct:free) أُلغي من OpenRouter. ظهر الخطأ في celery logs، فتم الانتقال إلى qwen3-next-80b-a3b-instruct المدفوع. لأن اسم النموذج محفوظ في DB، بقي التغيير محصوراً في صف واحد ودون downtime.
  • Sara model selection: اختبرنا DeepSeek R1، وظهر أنه يستهلك max_tokens في "thinking chain" داخلي ثم يعيد محتوى فارغاً. لذلك انتقلنا إلى DeepSeek V3.2 instruct، وهو أكثر ثباتاً لهذا الدور.
  • Cross-tab synchronization: تشغيل مراجعة طلب من تبويب واحد يبث الرسائل لكل التبويبات المشتركة في group application:<id>. كانت التبويبات الأخرى تحتاج حدثاً يمسح الحالة القديمة، لذلك أضفنا review_started قبل بدء الرسائل.
  • Alpine x-init guard: كان Alpine 3 يستدعي init() تلقائياً، ومع x-init="init()" كان التسجيل يحدث مرتين. حذفنا x-init وأضفنا _initDone idempotency guard لمنع تكرار الرسائل.
// FAILOVER PRIORITY

ترتيب الإنتاج المقترح هو: OpenRouter → Anthropic Direct → Groq → Ollama-local. بما أن الـ engine يعتمد على base_url قابل للتغيير، يتم التحويل بين المزودين عبر env var، لا عبر تعديل الكود.

// SECURITY & COMPLIANCE

الأمان والامتثال

يراعي التصميم متطلبات الأمن السيبراني البنكي، ونظام حماية البيانات الشخصية، وحساسية بقاء البيانات داخل المملكة.

طبقات الأمان السبع · Seven Security Layers

  1. Encryption in Transit: TLS 1.3 لكل HTTP و WebSocket. الشهادات من Let's Encrypt أو من CA محلي معتمد.
  2. Encryption at Rest: تشفير PostgreSQL عبر AWS KMS أو Azure Key Vault، مع إدارة المفاتيح من خلال HSM.
  3. Authentication & RBAC: Keycloak أو Azure AD مع MFA إلزامي للواجهة. الأدوار الأساسية: Loan Officer, Credit Reviewer, Compliance Auditor, Admin.
  4. Network Segmentation: الـ stack داخل VPC مخصّص. قاعدة البيانات بلا اتصال مباشر بالإنترنت، وLLM calls تمر عبر egress proxy مراقب، واتصالات SIMAH / Absher / Sakani عبر private link / VPN معتمد.
  5. Audit Logging: كل قرار AI، وكل تغيير في DB، وكل authentication event يكتب في append-only log داخل S3 Object Lock مع retention لمدة ٧ سنوات.
  6. Secrets Management: HashiCorp Vault أو AWS Secrets Manager. لا secrets داخل environment variables، ولا داخل الكود، ولا داخل DB.
  7. Incident Response: تنبيهات PagerDuty عند LLMUnavailable burst > ٥/دقيقة، أو authentication failure > ١٠/دقيقة، أو cost spike > ٢٠٠٪ من baseline.

PDPL — حماية بيانات العملاء

السؤال الحساس هنا واضح: ما الذي يصل فعلاً إلى النماذج الخارجية؟ التصميم يخفف المخاطر عبر ثلاث طبقات:

// 01

Data Minimization

لا يُمرر رقم الهوية الكامل أو رقم الجوال أو IBAN إلى النموذج. ما يمرر فقط هو القيم المُصنَّفة المطلوبة للقرار: SIMAH score (رقم)، DBR (نسبة)، نتيجة Absher (passed/flagged)، مسار التقييم (sakani/independent)، LTV (نسبة).

// 02

Pseudonymization

يُستبدل اسم المتقدّم باسم مستعار مثل "أبو محمد" قبل بناء البرومبت. الاسم الحقيقي ورقم الهوية يبقيان داخل DB البنك / LOS، ويرتبطان بالطلب عبر application_id فقط.

// 03

On-Prem Option

للبنوك الأكثر تحفظاً، يمكن تشغيل Ollama + Qwen3 على GPU محلي، بحيث لا تغادر بيانات SIMAH ولا وثائق العميل بيئة البنك. القرار النهائي يعتمد على مستوى المخاطر المقبول.

// CURRENT STATUS

المرحلة الحالية هي pilot وليست نسخة إنتاج مرخصة من ساما، والتوصية النهائية binding=False أي أنها مساعدة قرار وليست قراراً ملزماً. طبقات الأمان أعلاه تمثل هدف النسخة الإنتاجية. أما الـ MVP الحالي داخل docker-compose فلا يتضمن بعد MFA أو audit immutability أو secrets vault أو ربط حقيقي بـ SIMAH/Absher (يستخدم mock compliance snapshot)، وهذه الفجوات موثقة ومجدولة في خارطة الطريق.

القياس الأفقي · Horizontal Scale Plan

  • Celery Workers: التوسع من ٤ إلى ١٦ workers لمعالجة ما يقارب ٨٠٠ طلب/ساعة/cluster.
  • PostgreSQL: read replicas + PgBouncer connection pooling لأن الـ AgentReview write-heavy.
  • Redis: cluster mode للـ pub/sub broadcast على آلاف ApplicationConsumer متزامنة.
  • LLM API: نقطة الضغط الأهم، لكن تعدد المزودين عبر OpenRouter يقلل خطر التوقف عند مزود واحد.
  • Kubernetes orchestration: EKS Riyadh / AKS Riyadh مع HPA على CPU + queue depth.

// OBSERVABILITY & AUDIT

المراقبة والتدقيق

تخدم بيانات المراقبة ثلاث جهات: المهندسين، والمدققين، وموظفي القروض. كل جهة ترى ما تحتاجه من نفس السجل التشغيلي.

الإصدار الحالي (Pilot)

  • Application Logs: Django logging إلى Docker stdout، مع سجلات مستقلة لـ Celery workers، وتسجيل حالة كل LLM call ولكل Compliance API call (mock).
  • Database: كل AgentReview مؤرخ ويحمل structured + compliance_snapshot. يمكن فحص التوصيات من Django admin.
  • Diagnostic Console: الواجهة تطبع [Munjiz] msg received... في DevTools console، وهذا مفيد أثناء التشخيص الحي.

إصدار الإنتاج (Production)

  • Metrics: Prometheus + Grafana لقياس agent latency p50/p95/p99، ونسبة نجاح LLM calls، وتكلفة المراجعة لكل طلب، وتوزيع التوصيات (initial / conditional / human / not_recommended).
  • Logs: Loki للسجلات المنظمة والقابلة للبحث عبر LogQL. Retention لمدة ٣٠ يوماً hot و٧ سنوات cold للتدقيق.
  • Traces: OpenTelemetry يتتبع كل طلب من POST /trigger إلى Celery ثم Engine ثم LLM ثم DB، لتشخيص المشاكل العميقة على مستوى application_id.
  • Audit Trail: كل توصية AI وكل تغيير DB يدخل Kafka stream ثم S3 Object Lock مع 7-year retention وسجل immutable.
  • Cost Monitoring: تجميع يومي لكل tenant من response.usage، مع تنبيه عند تجاوز الميزانية لكل ١,٠٠٠ طلب.
  • Compliance Dashboard: لوحة للمدققين تعرض عدد التوصيات حسب النوع، نسب الحالات المصعدة للمراجع البشري، حالات override اليدوي، وتوزيع SIMAH/DBR/LTV bands.
// EXPLAIN-MY-DECISION

من منظور الحوكمة، يستطيع المدقق فتح "Decision Explainer" لأي قرار AI ورؤية: (١) السياق الكامل الذي رآه الوكيل، (٢) البرومبت النظامي وقتها، (٣) raw API response من LLM، (٤) ردود الوكلاء الآخرين التي بُني عليها القرار، (٥) النموذج المستخدم وإصداره. بهذا يصبح القرار قابلاً للفهم والمراجعة، لا مجرد نتيجة نهائية.

// ROADMAP

خارطة الطريق

تبدأ خارطة الطريق بإثبات القيمة على بنك واحد، ثم التوسع إلى عدة بنوك، وبعدها إلى منصة إقليمية متعددة المنتجات (تمويل عقاري، تمويل شخصي، SME).

Y1

السنة الأولى — Pilot & Proof

  • Shadow Mode على ١,٠٠٠ طلب لمدة شهرين: الوكلاء يراجعون ويصدرون توصيات، لكن قرار البنك يبقى يدوياً، ويُقاس مدى تطابق التوصية مع القرار البشري.
  • Limited Authority: تطبيق التوصية تلقائياً للحالات الواضحة (initial_approval / not_recommended) ودفع الباقي للمراجع البشري.
  • إثبات ROI بأرقام فعلية على بنك واحد، ثم توقيع عقد إنتاج.
  • تجهيز ملف SAMA dossier وتقديمه (التوصية ليست قراراً ملزماً).
Y2

السنة الثانية — Multi-Tenant Rollout

  • ربط حقيقي بـ SIMAH gateway و Absher/Yakeen و Sakani.
  • Multi-tenancy لاستقبال البنك الثاني والثالث على نفس الـ stack.
  • قنوات جديدة: تطبيق الموبايل، الإنترنت البنكي، broker portals.
  • إضافة وكلاء جدد: Compliance Agent للـ auto-AML check، Restructuring Agent لإعادة هيكلة التمويل القائم.
Y3

السنة الثالثة — Regional Lending SaaS

  • منصّة SaaS متعدّدة البنوك في المملكة لمنتجات الإقراض المختلفة.
  • توسعة إقليمية إلى الإمارات ومصر والكويت، مع تكييف الفحوصات (Etihad Credit Bureau، I-Score…) ونبرة مناسبة لكل سوق.
  • توسعة المنتج: تمويل شخصي، تمويل MSME، تمويل سيارات.
  • Marketplace للوكلاء، بحيث تستطيع البنوك تركيب وكلاء طرف ثالث مثل anti-fraud أو climate-risk advisor.
  • API عام للـ enterprise customers.
