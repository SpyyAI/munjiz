System Architecture

// 01 · COMPONENTS & FLOW

Munjiz is a hierarchical multi-agent system. A central Orchestrator coordinates four specialist agents over an A2A (agent-to-agent) protocol. Every agent has access to internal tools, queries LLM APIs through OpenRouter, and writes each reasoning step into memory — Redis for live context, PostgreSQL for an immutable, replayable audit log.

A
// AGENTS
5 Specialized Agents
Orchestrator + Sara, Fahad, Khalid, Nora — each a domain expert with its own LLM and prompt.
T
// TOOLS
Internal APIs
Branch ops, queue snapshot, WhatsApp notifier, analytics, customer / VIP lookup.
L
// LLM APIS
OpenRouter Gateway
Routes to Claude Opus / Sonnet, Qwen3, DeepSeek and GPT-4o mini via one OpenAI-compatible endpoint.
M
// MEMORY
Redis + PostgreSQL
Redis caches live context; Postgres stores every AgentMessage as an immutable audit log.
  1. TriggerManager picks a scenario or a live event hits the dashboard; the request travels over WebSocket to the Orchestrator.
  2. CoordinateOrchestrator dispatches sub-tasks to specialist agents over the A2A protocol, by domain.
  3. ReasonEach agent calls its LLM via OpenRouter, queries internal tools, and persists its reasoning to memory.
  4. Decide & StreamOrchestrator merges outputs, enforces authority limits, and streams the final decision back live.

What decisions it makes

// 02 · AUTHORITY MATRIX

Each agent acts inside a strict authority envelope. The Orchestrator enforces these limits before any action — auto-executing low-risk operations, proposing higher-stakes ones to the manager, and escalating anything compliance- or money-sensitive.

Decision Owner Authority
Re-route customer to a faster kiosk or counter Nora · Khalid Auto
Send WhatsApp wait-time notification Khalid Auto

Branch-wide SLA breach above threshold Orchestrator Escalate

Architecture type

// 03 · MULTI-AGENT
Munjiz is a Multi-Agent System (MAS) with a hierarchical orchestrator pattern over an A2A protocol — 1 orchestrator coordinating 4 specialized peers.
  • Why multi-agent — branch decisions span analytics, ops, customer care and queue planning; one prompt cannot reason them all coherently.
  • Why hierarchical — the Orchestrator enforces authority limits and produces one consistent decision instead of conflicts.
  • Why A2A — agents share structured context, debate when needed, and produce an immutable, replayable trail.
  • Why heterogeneous LLMs — each agent uses the model best suited to its job.
