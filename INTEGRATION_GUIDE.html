إخلاء المسؤولية حول أسماء المنتجات

دقّة هندسية: نُميِّز بوضوح بين ما هو مُؤكَّد عن منتجات Beyond Technology وما هو فئات منتجات بنكية مَعروفة نُسقطها على Beyond بشكل معقول.

// IMPORTANT — READ BEFORE PROCEEDING

Beyond Technology هي شركة سعودية متخصّصة في حلول الفروع البنكية. منتجاتها المنشورة تَشمل فئات قياسية في القطاع: Queue Management, Customer Journey Analytics, Self-Service Channels, Workforce Management, Digital Signage, Mobile Banking. هذه الوثيقة تَستخدم تَسميات الفئات (Categories) كنقاط تَكامُل، وليست SKUs تجاريّة دقيقة من Beyond.

كل integration spec هنا قابل للتنفيذ على المنتجات الفعلية في كل فئة. الـ exact API endpoints, schema versions, و authentication tokens سَتُؤكَّد في Integration Kickoff Workshop مع فريق Beyond التقني.

Beyond Technology operates Saudi-market branch-banking solutions across well-known categories. This document uses category-level names as integration anchors, not specific Beyond commercial SKUs. Every integration spec is implementable across actual products in each category. Exact API endpoints, schema versions, and auth tokens will be confirmed in the integration kickoff workshop.

02
// THE REAL MANAGER'S PAIN

آلام مدير الفرع الواقعية

قبل أي حديث عن المعمارية، نَستعرض ٥ آلام يومية حقيقية لمدير فرع — كل واحدة مَوثَّقة بأرقام واقعية، وكل واحدة لها مَصدر بيانات فعلي يَأتي من منظومة Beyond.

الألم الواقعي (وليس "dummy")

مدير فرع البنك الحقيقي ليس "شخصية ديمو". إنه إنسان يَتعامل مع ٥٠+ قراراً غير-استراتيجي يومياً، أغلبها يَستهلك ٢-١٠ دقائق من وقته، ويَتسرَّب أحياناً إلى أخطاء تَكلِّف البنك مالاً أو تَكسر علاقة مع عميل ذو قيمة عالية. منجِز لا يَستهدف الأمور الكبيرة (التي يَجب أن يَفعلها المدير) — بل الأمور الصغيرة المُتكرِّرة التي تَستنزف وقته.

01

"العميل المميَّز يَنتظر، وأنا لا أعلم"

الواقع: العميل أبو محمد السديري (VIP، ٨.٥ مليون ريال قيمة سنوية) يَدخل الفرع. الموظف على البوابة لا يَعرفه شخصياً. يَأخذ تذكرة عادية، يَنتظر ١٢ دقيقة. يَخرج غاضباً.

أين تَفشل الأنظمة الحالية: Beyond QMS لديه التذكرة، Core Banking لديه ملف العميل، لكن لا يوجد ربط لحظي. المدير يَكتشف بعد فوات الأوان من تقرير NPS.

كيف يَحلّها منجِز: الوكيل خالد يَقرأ ticket ID من Beyond QMS لحظياً، يُخصِّبه بـ customer profile من Core Banking، يَكتشف VIP، يُنبِّه المنسق فوراً، المنسق يَتَّخذ قراراً تلقائياً (لينا تَستقبله شخصياً + قهوة + تَجاوز الطابور).

02

"الموظف في استراحة وأنا في ذروة"

الواقع: الساعة ١١:٣٠ خميس نهاية شهر. ٣٤ عميل بالطابور. خالد الغامدي (الصراف الأسرع) في استراحته الـ ٣٠-دقيقة المُقَرَّرة. لا أحد يُذكِّر المدير أن الذروة تَأتي.

أين تَفشل الأنظمة الحالية: Beyond WFM لديه جدول الاستراحات. Beyond Insights لديه أنماط الذُّرى التاريخيّة. لكن أحدهما لا يَسأل الآخر "هل الاستراحة الآن مناسبة بالنظر للذروة المُتوقَّعة؟".

كيف يَحلّها منجِز: سارة (Analytics) تَتَنبَّأ بالذروة قبل ساعة من خلال مزج بيانات WFM + Insights + Salary Calendar. فهد (Operations) يُحَذِّر: "خالد في استراحة، الذروة تَبدأ خلال ١٠ دقائق". المنسق يُرسل push notification لخالد عبر Beyond WFM API: "نَحتاجك ٢٠ دقيقة قبل، نَعَوِّضك بـ ١٥ دقيقة إضافيّة لاحقاً".

03

"عبد الرحمن غضبان وأنا لا أعرف"

الواقع: العميل عبدالرحمن العمري (Premium، ٦٤٠ ألف ريال) قَدَّم طلب قرض عقاري الجمعة الماضية. اليوم خميس، لم يَتلَقَّ رداً. يَأتي للفرع غاضباً، يَطلب إغلاق حسابه. المدير لا يَعرف عن المشكلة حتى يَسمَع الصراخ.

أين تَفشل الأنظمة الحالية: Loan Origination System لديه الحالة. Beyond Voice (NPS) لم يَكتشفه بعد. Beyond Journey لديه نمط الزيارات لكن لم يُربط بـ مشاعر العميل.

كيف يَحلّها منجِز: سارة تَكتشف الذُّخر السلبي (CSAT droppen، يَتَّصل ٣ مرّات لـ call center، يَقرأ بريداً سَلبياً) قبل وصوله. خالد يَستلم الـ alert عند دخوله، يَجلس مع لينا (مستشارة العلاقات) جاهزة. المنسق يُحَضِّر تَعويضاً مُسبَقاً (٥٠٠ ريال ضمن صلاحيته، أو يُصَعِّد للمدير ماجد لتعويض أكبر).

04

"وقت الصلاة قَرُب وعندي ٢٢ عميل"

الواقع: أذان الظهر ١٢:١٥. الفرع يَحتاج إغلاقاً مؤقَّتاً ٢٥ دقيقة. ٢٢ عميل بالطابور. مَن يَخدم؟ مَن يَعتذر منه؟ كيف نُبَلِّغ بدون إثارة غضب؟

أين تَفشل الأنظمة الحالية: Beyond QMS يُغلَق يدوياً. Beyond Signage يُحَدَّث يدوياً. Beyond Communication (WhatsApp/SMS) يُرسل يدوياً. كل ذلك على المدير في ٥ دقائق.

كيف يَحلّها منجِز: نورة (Queue) تَحسب: "نَستطيع خدمة ١٢ عميل قبل الصلاة، نُؤجِّل ١٠ بإشعار". المنسق يُنفِّذ تلقائياً عبر Beyond APIs: (أ) يُحَدِّث Beyond Signage لعَرض عدّاد تنازلي، (ب) يُرسل WhatsApp جماعي عبر Beyond Communication لكل عميل في الطابور، (ج) يُحَدِّد ٣ كيوسكات تَبقى شغّالة، (د) يُحَضِّر بروتوكول إعادة الفتح ١٢:٤٠.

05

"المدير المحلي في اجتماع، والمنطقة تَطلَب تَقريراً"

الواقع: آخر اليوم، المدير الإقليمي يَطلَب: "كم خَدَمنا اليوم؟ كم تَجاوُز SLA؟ كم VIP استَقبَلنا؟ ما الدروس؟". المدير المحلي يَفتَح ٣ أنظمة، يَكتُب يدوياً، يَأخذ ٤٥ دقيقة.

أين تَفشل الأنظمة الحالية: Beyond Insights لديه الأرقام، لكن لا أحد يَكتب الـ narrative الذي يَفهَمه المدير الإقليمي.

كيف يَحلّها منجِز: عند نهاية اليوم، المنسق يَستَدعي كل وكلائه ويَطلُب ملخّصاً. سارة تَكتب الأرقام، فهد يَكتب الموارد، خالد يَكتب رضا العملاء، نورة تَكتب الدروس. المنسق يَدمج كل ذلك في تقرير عربي رفيع الجودة، ويُرسله بالبريد للمدير الإقليمي عبر Beyond Insights API. المدير المحلي يَعتمده بنقرة واحدة.

قنوات التَكامُل · Integration Channels

// CHANNEL 01

REST API (Read)

منجِز يَقرأ من Beyond products عبر REST endpoints معيارية (OpenAPI 3 spec). frequency: on-demand لكل سيناريو + كاش ٣٠ ثانية للأرقام السريعة.

FORMAT: JSON · AUTH: OAuth2
// CHANNEL 02

Webhooks (Push)

Beyond products تُرسل webhooks لمنجِز عند أحداث: ticket_taken, customer_arrived_vip, kiosk_maintenance, sla_breach_imminent. منجِز يَستجيب بقرار فوري.

FORMAT: JSON · DELIVERY: at-least-once
// CHANNEL 03

Action API (Write)

منجِز يَكتب إلى Beyond products لتنفيذ القرارات: redirect_ticket, update_signage_content, request_break_delay, send_whatsapp_blast.

FORMAT: JSON · AUTH: Service Account + RBAC
// CHANNEL 04

Event Stream (Optional)

للبنوك ذات الحجم الكبير، Kafka topic مشترك بين Beyond و منجِز. أحداث lossless مع replay capability.

PROTOCOL: Kafka · RETENTION: 7 days

// SECURITY & TRUST BOUNDARIES

الأمان وحدود الثقة

حدود الأمان بين Munjiz و Beyond، نموذج RBAC، تَدفُّق البيانات الحَسَّاسة، الالتزام بساما + PDPL.

نَموذج الثقة · Trust Model

Munjiz هي service account داخل بيئة البنك — ليست طرفاً ثالثاً خارجياً. تَعمل داخل نفس الـ VPC أو datacenter، تَستخدِم الـ identity provider الخاص بالبنك (Active Directory أو Keycloak)، وتُعامَل كأي خدمة داخلية أخرى.

// BOUNDARY 01

Munjiz ↔ Beyond (intra-VPC)

اتصال داخل VPC البنك. mTLS مَطلوب على كل service-to-service call. شهادات صادرة عن Bank Internal CA. منصب Beyond لا يَخرج، منصب Munjiz لا يَخرج.

// BOUNDARY 02

Munjiz ↔ LLM Provider (egress)

عبر egress proxy مُراقَب فقط. PII removal layer تَمنع تَسرُّب أرقام حسابات/هويّات/أرقام جوّال. الأسماء تُستَبدَل بـ aliases. البيانات الحَسَّاسة مُجَمَّعة (segment, value bracket).

// BOUNDARY 03

User ↔ Munjiz UI

RBAC إجباري عبر Bank's IDP. أربع أدوار: Branch Manager (full), Operations Lead (read), Auditor (read+audit_log), Admin (config). MFA إجباري لكل login.

تَدفُّق البيانات الحَسَّاسة · Sensitive Data Flow

// Customer C001 data flow through Munjiz

[CBS]  name=محمد بن عبدالعزيز السديري, ID=1010..., phone=+9665...
        ↓

أنماط الفشل والتسوية

ما الذي يَحدث عندما تَفشل أيّ عقدة في النظام — Beyond، أو CBS، أو Munjiz نفسه — وكيف نَستعيد بدون فقد بيانات أو قرارات معلَّقَة.

الفشل · Failureالكشف · Detectionالاستجابة · Response
Beyond QMS unreachable Circuit breaker: 5 errors / 30s Sara, Fahad, Khalid skip queue_state field; Orchestrator gets queue_state: null + integrations.qms.status: "degraded" in context. Decisions explicitly avoid queue-routing actions.

Action write fails (e.g. signage update) HTTP error from Beyond Idempotent retry 3× with exponential backoff. If still failing, action marked failed in audit, manager notified. Other actions in the decision continue.
LLM provider rate limit HTTP 429 SDK auto-retry 2× with backoff. If still failing, multi-provider failover (OpenRouter → Anthropic Direct → Groq → Ollama-local).

Munjiz cluster failure Health probe / heartbeat K8s reschedules pods. WebSocket clients reconnect automatically (built-in). In-flight scenarios are abandoned (their decisions never persist), users see no fake state.
Authority matrix change Bank's auth service notifies Munjiz refreshes per-branch authority matrix; in-flight decisions complete with old matrix; new decisions use updated matrix immediately.
Customer object out of sync (Beyond Journey ≠ CBS) Reconciliation flags mismatch Khalid prefers CBS as source-of-truth for value/segment, Beyond Journey for behavior patterns. Conflict logged; data quality team alerted.

خطّة الترحيل من الأنظمة الحالية

كيف يَنتقِل بنكٌ من "كل قرار يدوي + أنظمة معزولة" إلى "Munjiz يُنَسِّق Beyond suite" — في ٤ مَراحل بدون انقطاع خِدمة.

01

المرحلة الأولى — Shadow Mode (شهرين)

الهدف: منجِز يَقرأ Beyond بدون كتابة. يُسَجِّل القرارات التي كان سَيَتَّخذها في audit log منفصل.

  • تَفعيل read-only service account لمنجِز على كل Beyond products
  • منجِز يُشَغِّل ٢-٣ سيناريوهات/فرع/يوم في وضع silent
  • أسبوعياً: مقارنة قرارات منجِز مع قرارات المدير الفعلية → دقّة ٨٥٪+ هي الـ gate
  • صفر تَأثير على الإنتاج. صفر مَخاطر.
02

المرحلة الثانية — Limited Authority (٣-٤ شهور)

الهدف: منجِز يَكتُب لـ Beyond، لكن فقط actions منخفضة المخاطر.

  • السماح بـ: signage updates, kiosk routing, WhatsApp blasts
  • المنع من: financial actions, staff actions, customer segment changes
  • كل action تَمرّ عبر approval queue للمدير لمدّة أسبوعَين، ثم تَنتقل لـ auto
  • قياس: SLA compliance يَتَحَسَّن ١٠٪+، شكاوى تَنخَفِض، رضا المدير يَرتَفِع
03

المرحلة الثالثة — Full Authority within Limits (٤-٦ شهور)

الهدف: منجِز يَتَصَرَّف ضمن صلاحيات المنسق المعتَمَدَة.

  • تَفعيل: staff break delays, customer compensation ≤500 SAR, ticket escort
  • التَصعيد للمدير فقط للـ exceptions الحقيقيّة (تعويضات أكبر، تمديد ساعات)
  • توسعة من ٣-٥ فروع إلى ٢٠-٣٠
  • قياس: ٨٥٪ من القرارات تلقائية، المدير يُفَرِّغ ٣ ساعات/يوم للاستراتيجية
04

المرحلة الرابعة — Network-Wide + Continuous Learning (٦+ شهور)

الهدف: كل الفروع، مع تَعَلُّم مُستَمر من القرارات الفعلية.

  • إطلاق على ٢٠٠+ فرع
  • Central Observability dashboard للمدير الإقليمي
  • A/B testing بين تَكوينات النماذج
  • Continuous prompt refinement من الـ overrides اليدوية
  • إضافة وكلاء جُدُد: Compliance Agent (auto-AML), Sales Agent (cross-sell)
